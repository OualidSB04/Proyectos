Programación de servicios y procesos

La programación multiproceso consiste en crear y gestionar varios procesos que se ejecutan de forma independiente. Cada proceso tiene su propio espacio de memoria aislado, lo que evita interferencias pero obliga a usar mecanismos de comunicación entre procesos (IPC) como colas, tuberías o memoria compartida. El sistema operativo planifica los procesos asignándoles tiempo de CPU. Un proceso atraviesa varios estados: nuevo, listo, en ejecución, bloqueado y terminado. La planificación busca maximizar el uso del procesador y reducir los tiempos de respuesta.

La programación multihilo permite que varias tareas se ejecuten dentro del mismo proceso, compartiendo su memoria. Los hilos son más ligeros que los procesos y la comunicación entre ellos es directa porque comparten variables. Sin embargo, ese acceso compartido provoca condiciones de carrera cuando dos hilos modifican el mismo dato a la vez. Para evitarlo se usan mecanismos de sincronización como los cerrojos (locks), los semáforos y las operaciones atómicas. Un cerrojo garantiza la exclusión mutua: solo un hilo entra a la sección crítica cada vez. Un mal diseño de la sincronización puede provocar interbloqueos (deadlocks), donde varios hilos se esperan mutuamente para siempre.

La programación de comunicaciones en red se basa en el modelo cliente-servidor. El servidor abre un socket, escucha en un puerto y acepta conexiones entrantes. El cliente se conecta a la dirección y el puerto del servidor para intercambiar datos. Los sockets TCP ofrecen una conexión fiable y ordenada, mientras que UDP es más rápido pero sin garantías de entrega. Un servidor concurrente atiende a varios clientes a la vez, normalmente dedicando un hilo o un proceso a cada conexión. Los protocolos de aplicación definen el formato de los mensajes que se intercambian.

La generación de servicios en red se apoya en los servicios web, que exponen funcionalidades a través de Internet de forma estandarizada. Una API REST organiza el acceso a recursos mediante URLs y los métodos HTTP: GET para leer, POST para crear, PUT para actualizar y DELETE para borrar. Los datos suelen intercambiarse en formato JSON. El servidor responde con códigos de estado que indican el resultado: 200 correcto, 201 creado, 404 no encontrado, 500 error interno. Un contrato bien documentado, por ejemplo con OpenAPI, permite que desarrolladores externos consuman el servicio sin ambigüedades.

La programación segura aplica técnicas para proteger los datos y las comunicaciones. El hasheado convierte una contraseña en un resumen irreversible; para almacenarla con seguridad se añade un salt aleatorio y se usan funciones lentas como PBKDF2, bcrypt o Argon2, que dificultan los ataques por fuerza bruta. El cifrado simétrico, como AES, protege la confidencialidad usando la misma clave para cifrar y descifrar. El cifrado autenticado, como AES-GCM, añade además integridad: si alguien altera el mensaje cifrado, el descifrado falla. La aleatoriedad criptográfica debe provenir de un generador seguro (CSPRNG), nunca de funciones como rand o Math.random. Los protocolos seguros como TLS combinan cifrado, autenticación e integridad.

La recuperación aumentada por generación, conocida como RAG, combina una base de datos vectorial con un modelo de lenguaje. Primero se trocea un texto grande en fragmentos llamados chunks. Cada chunk se transforma en un vector numérico mediante un modelo de embeddings que captura su significado. Esos vectores se almacenan en una base de datos vectorial como ChromaDB. Cuando el usuario hace una pregunta, esta también se convierte en un vector y la base de datos devuelve los chunks cuyo significado es más cercano, midiendo la similitud semántica. Finalmente, un modelo de IA recibe esos fragmentos como contexto y redacta una respuesta natural y comprensible para la persona.
